The Silver layer applies data quality rules, standardization, and cleansing transformations to raw Bronze data. This layer ensures data consistency and prepares data for business consumption.

## Transformation Types

### Deduplication

Removes duplicate records using window functions to keep only the most recent version:

```sql
SELECT *, 
  ROW_NUMBER() OVER (
    PARTITION BY cst_id 
    ORDER BY cst_create_date DESC
  ) AS flag_last
FROM bronze.crm_cust_info
WHERE flag_last = 1;
```

<Info>
  This pattern ensures only the latest customer record is retained based on the creation date.
</Info>

### Code Standardization

Converts abbreviated codes into human-readable values:

<Tabs>
  <Tab title="Gender Codes">
    ```sql
    CASE upper(trim(cst_gndr)) 
      WHEN 'M' THEN 'Male' 
      WHEN 'F' THEN 'Female' 
      ELSE 'n/a'
    END as cst_gndr
    ```
  </Tab>
  
  <Tab title="Marital Status">
    ```sql
    CASE upper(trim(cst_marital_status)) 
      WHEN 'S' THEN 'Single' 
      WHEN 'M' THEN 'Married' 
      ELSE 'n/a'
    END as cst_marital_status
    ```
  </Tab>
  
  <Tab title="Product Line">
    ```sql
    CASE upper(trim(prd_line))
      WHEN 'M' THEN 'Mountain'
      WHEN 'R' THEN 'Road'
      WHEN 'S' THEN 'Other sales'
      WHEN 'T' THEN 'Touring'
      ELSE 'n/a'
    END as prd_line
    ```
  </Tab>
  
  <Tab title="Country Codes">
    ```sql
    CASE 
      WHEN trim(cntry) IN ('USA','US') THEN 'United States'
      WHEN trim(cntry) = 'CAN' THEN 'Canada'
      WHEN trim(cntry) = 'DE' THEN 'Germany'
      WHEN trim(cntry) = '' OR trim(cntry) IS NULL THEN 'n/a'
      ELSE trim(cntry)
    END as cntry
    ```
  </Tab>
</Tabs>

### Date Formatting

Converts integer date values (YYYYMMDD format) to proper DATE types:

```sql
CASE
  WHEN sls_ord_dt IS NULL
    OR sls_ord_dt = 0
    OR length(sls_ord_dt::text) != 8
  THEN NULL
  ELSE to_date(TRIM(sls_ord_dt::text), 'YYYYMMDD')
END AS sls_ord_dt
```

<Warning>
  Invalid dates (null, zero, or incorrect length) are converted to NULL to prevent data quality issues.
</Warning>

### Data Validation and Correction

#### Sales Amount Validation

Ensures sales amounts are consistent with quantity and price:

```sql
CASE 
  WHEN sls_sales IS NULL 
    OR sls_sales <= 0 
    OR sls_sales != sls_quantity * abs(sls_price) 
  THEN sls_quantity * abs(sls_price)
  ELSE sls_sales
END as sls_sales
```

#### Price Correction

Handles negative or null prices:

```sql
CASE 
  WHEN sls_price IS NULL 
    OR sls_price < 0 
    OR sls_price = 0 
  THEN sls_sales / abs(sls_quantity)
  WHEN sls_price < 0 
  THEN abs(sls_price)
  ELSE sls_price
END as sls_price
```

### Data Enrichment

#### Category Extraction

Extracts category ID from product key:

```sql
SELECT 
  prd_id,
  replace(substring(trim(prd_key), 1, 5), '-','_') as cat_id,
  substring(prd_key, 7, length(prd_key)) as prd_key,
  ...
FROM bronze.crm_prd_info
```

#### Product End Date Calculation

Uses LEAD window function to calculate product end dates:

```sql
lead(prd_start_dt) OVER (
  PARTITION BY prd_key 
  ORDER BY prd_start_dt
) - 1 as prd_end_dt
```

#### Customer ID Normalization

Removes prefixes and standardizes customer IDs:

```sql
CASE 
  WHEN cid LIKE 'NAS%' THEN substring(cid, 4, length(cid))
  ELSE cid
END as cid
```

## Transformation Examples

### Customer Information Transformation

<CodeGroup>
```sql Before (Bronze)
SELECT 
  cst_id,
  cst_key,
  cst_firstname,
  cst_gndr,        -- 'M' or 'F'
  cst_marital_status  -- 'S' or 'M'
FROM bronze.crm_cust_info;
```

```sql After (Silver)
SELECT
  cst_id, 
  trim(cst_key), 
  trim(cst_firstname) as cst_firstname,
  CASE upper(trim(cst_gndr)) 
    WHEN 'M' THEN 'Male' 
    WHEN 'F' THEN 'Female' 
    ELSE 'n/a'
  END as cst_gndr,  -- 'Male' or 'Female'
  CASE upper(trim(cst_marital_status)) 
    WHEN 'S' THEN 'Single' 
    WHEN 'M' THEN 'Married' 
    ELSE 'n/a'
  END as cst_marital_status  -- 'Single' or 'Married'
FROM bronze.crm_cust_info;
```
</CodeGroup>

### Sales Details Transformation

<CodeGroup>
```sql Before (Bronze)
-- Dates stored as integers: 20240315
sls_ord_dt INT,
sls_ship_dt INT,
sls_sales INT,
sls_price DECIMAL(10, 2)
```

```sql After (Silver)
-- Dates converted to DATE type
to_date(TRIM(sls_ord_dt::text), 'YYYYMMDD') AS sls_ord_dt,
to_date(TRIM(sls_ship_dt::text), 'YYYYMMDD') AS sls_ship_dt,
-- Sales amount validated
sls_quantity * abs(sls_price) AS sls_sales,
-- Price corrected
abs(sls_price) AS sls_price
```
</CodeGroup>

## Loading Process

The `silver.load_silver()` procedure executes transformations:

<Steps>
  <Step title="Truncate Target Table">
    Each Silver table is truncated before loading
  </Step>
  
  <Step title="Execute Transformation">
    INSERT INTO statement with transformation logic applied
  </Step>
  
  <Step title="Log Progress">
    Execution time and status logged for each table
  </Step>
  
  <Step title="Handle Errors">
    Any failures trigger exception with detailed error message
  </Step>
</Steps>

```sql
CALL silver.load_silver();
```

<Note>
  The procedure processes all six tables sequentially with comprehensive error handling and performance logging.
</Note>
